**قرار سياسة الاسترداد (مسودة — تنتظر الاعتماد النهائي من إدارة العمل #8):**

1. **نافذة الاسترداد (Refund Window)**
   - **دفعات زائدة / تسوية حساب:** يُسمح بالاسترداد الكامل خلال **14 يومًا** من تاريخ الدفع إذا ثبت أن المبلغ المدفوع يتجاوز المبلغ المستحق.
   - **خدمات لم تُقدَّم:** يُسمح بالاسترداد الكامل خلال **48 ساعة** من موعد الخدمة المقررة إذا لم يتم تقديم الخدمة لسبب يتحمله المركز (إلغاء من المركز، عدم توفر الطبيب/الجهاز، إلخ).
   - **خدمات تم تقديمها:** لا يُسترد المبلغ بعد تقديم الخدمة إلا في حالات استثنائية تتطلب موافقة الإدارة.

2. **الاسترداد الجزئي (Partial Refund)**
   - يُسمح بالاسترداد الجزئي عند إلغاء جزء من الخدمات المدفوعة مسبقًا قبل تنفيذها.
   - يُحسب المبلغ المسترد بناءً على تكلفة الخدمات المُلغاة فقط، مع خصم أي رسوم إدارية بنسبة **0%** (مسودة قابلة للتعديل حسب قرار الإدارة).
   - لا يُسمح بالاسترداد الجزئي للأدوية المُستهَلَكة أو للفحوصات التي تم إجراؤها.

3. **الشحن backs (Chargebacks)**
   - عند استلام طلب chargeback من البنك/بوابة الدفع:
     - يتم تعليق الحالة ماليًا حتى انتهاء التحقيق.
     - يُرفق إثبات الخدمة (فاتورة، تقرير طبي، إيصال) مع الرد على البنك.
     - إذا قُبل الرد، يُغلق الطلب ولا يُسترد المبلغ.
     - إذا رُفض الرد، يُسجل الاسترداد كـ `chargeback_loss` في حسابات المركز.
   - رسوم Chargeback تتحملها إدارة المركز ولا تُحمل المريض.

4. **طرق الاسترداد**
   - يتم الاسترداد بنفس طريقة الدفع الأصلية whenever possible.
   - الاسترداد النقدي يقتصر على المبالغ المدفوعة نقدًا فقط، ويتطلب توقيع المستفيد وإثبات هوية.
   - وقت معالجة الاسترداد: **3-5 أيام عمل** للدفع الإلكتروني، و**يوم عمل واحد** للدفع النقدي.

5. **الموافقات المطلوبة**
   - استرداد ≤ 500 ر.س.: موافقة مدير الصندوق أو المدير.
   - استرداد > 500 ر.س.: موافقة الإدارة المالية أو المدير العام.
   - يُسجل كل استرداد في سجل audit مع سبب الاسترداد ومرجع القرار.

6. **حالات غير قابلة للاسترداد**
   - الأدوية والمستلزمات المُستهَلَكة.
   - الفحوصات والتحاليل التي تم إجراؤها.
   - المواعيد الملغاة من المريض بعد مرور 24 ساعة على الموعد (no-show).
   - الاشتراكات والباقات السنوية بعد بدء الاستخدام.

  - Audit logger عام لكل الإشارات.
  - Notification subscriber factory يربط subset من الإشارات بـ `NotificationService.send_notification(...)`.

**عقد `NotificationService.send_notification`:**
- Producer: أي route أو service في النظام (غير موحّد).
- Payload: `(recipient_id | recipient_role | recipient_department_id, title, message, notification_type, sender_id, is_urgent, expires_at, template_name/template_variables)`.
- Delivery guarantee: best-effort synchronous داخل نفس DB session. إذا فشل الإرسال تُرجع `{success: False}` ولكن لا يوجد retry queue أو outbox pattern.
- Idempotency: **غير موجودة**. استدعاء مكرر ينشئ `Notification` مكرر.
- Ordering: حسب `sent_at` DESC.
- Persistence: جدول `notifications` مع `is_read`, `is_urgent`, `expires_at`.

**الفجوات الحالية:**
- لا يوجد event schema موحّد (كل signal يستخدم kwargs حرة).
- لا يوجد idempotency key.
- لا يوجد outbox/retry للإشعارات التي تفشل أثناء الطلب.
- بعض الإشعارات تُرسل مباشرة والبعض عبر signals — ازدواجية producer.
- لا يوجد consumer registry واضح.

**متطلبات UX1-003 (Unified Work Inbox):**
- يجب أن يُنتج كل workflow event (زيارة، طلب مختبر، أمر أشعة، وصفة، فاتورة، حالة طوارئ) صفًا في `Notification` مع `related_entity_type` و `related_entity_id` موحّدين.
- يجب دعم idempotency key (`producer_event_id`) قبل بناء Inbox.
- يجب توحيد producer pattern: Route/Service يرسل Signal → Subscriber يستدعي NotificationService (لا استدعاء مباشر من الـ routes).
- يجب إضافة outbox أو background retry قبل اعتبار الإشعارات موثوقة للعمليات الحرجة.

### S0-004: Combined Access Contract (Entitlement + Authorization) (معدّل)

| البند              | القيمة                                                                                                                                 |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| **الأولوية**       | S0                                                                                                                                     |
| **الهدف**          | تعريف combined access contract: Entitlement ليس Authorization. الـ resolver يجيب على "هل الدفع يغطي هذه الـ capability؟" فقط           |
| **المبدأ الأساسي** |                                                                                                                                        |
|                    | **Access = authenticated identity + active tenant + tenant entitlement + role permission + branch scope + resource ownership/context** |
|                    | EntitlementResolver لا يمنح capability سريرية لمجرد أن المستأجر دفع ثمنها                                                              |

**لائحة المسؤوليات:**

| المسؤولية                                            | المالك                                             | تدمج في EntitlementResolver؟ |
| ---------------------------------------------------- | -------------------------------------------------- | ---------------------------- |
| هل المستأجر نشط ومسدد؟                               | ✅ نعم                                              |                              |
| هل المستأجر مخوّل لهذه الـ module/capability؟        | ✅ نعم                                              |                              |
| هل يتجاوز الـ tenant حدوده (users/patients/storage)? | ✅ نعم                                              |                              |
| هل المستخدم لديه role/permission للـ action؟         | ❌ لا — نظام roles الحالي (`models/permissions.py`) |                              |
| هل الفرع (branch) مصرح له؟                           | ❌ لا — Branch system الحالي                        |                              |
| هل المستخدم يملك حق الوصول لهذا الـ patient/visit؟   | ❌ لا — Resource ownership logic الحالي             |                              |
| هل المستخدم platform-owner أم tenant-user؟           | ❌ لا — Owner module boundary الحالي                |                              |
| هل API key موسوم بـ scope؟                           | ❌ لا — API key scope الحالي                        |                              |

**Layered Enforcement:**

| الطبقة                | ما يُفحص                                | آلية التطبيق                                                                             |
| --------------------- | --------------------------------------- | ---------------------------------------------------------------------------------------- |
| **Route**             | Tenant entitlement + role permission    | Decorator composition: `@require_entitlement("lab")` + `@require_permission("lab.edit")` |
| **Service**           | Tenant entitlement + resource ownership | `resolver.assert_entitled()` + `ownership_check()`                                       |
| **Worker/Background** | Tenant entitlement                      | `resolver.assert_entitled(tenant_id, "notification")`                                    |
| **Frontend**          | Tenant entitlement + role               | `{% if is_entitled("lab") and has_permission("lab.edit") %}`                             |
| **Audit**             | كل denied attempt                       | `PlatformAuditLog` يسجل action, reason, actor, tenant                                    |

**Cache Strategy (معدّل):**

* **افتراضي آمن:** request-local memoization فقط (لكل request، يُحلّ مرة واحدة)
* Shared cache فقط بعد التحقق من: deployment topology, multi-worker behavior, invalidation, suspension propagation, emergency-disable
* **Suspension و feature revocation** يجب أن يسريا فورًا أو ضمن strict limit موثّق
* لا TTL 5 دقائق كافتراضي

### S0-005: Tenant Provisioning Lifecycle with Subscription Lines (معدّل)

| البند                 | القيمة                                                           |
| --------------------- | ---------------------------------------------------------------- |
| **الأولوية**          | S0                                                               |
| **الهدف**             | تعريف lifecycle كامل مع subscription line items (base + add-ons) |
| **المتطلبات المسبقة** | S0-004                                                           |
| **الحجم**             | L                                                                |

**Lifecycle States:**

```
SIGNUP (pending)
  │
  ▼
TRIAL (trial期限, line.trial_end)
  │
  ├──▶ ACTIVE (paid)
  │       │
  │       ├──▶ UPGRADE (line القديم ينتهي, line جديد يُنشأ — package_version_id مختلف, immutable)
  │       ├──▶ DOWNGRADE (line القديم ينتهي, line جديد يُنشأ — capabilities الجديدة فقط ممنوعة)
  │       │       └── البيانات التاريخية readable; الإنشاء فقط مقيد
  │       ├──▶ ADD-ON (line جديد category=addon)
  │       └──▶ RENEW (current_period_end يتمدّد)
  │
  ├──▶ EXPIRED (all lines expired + grace مرّ)
  │       └──▶ بيانات clinical/financial intact
  │
  ├──▶ SUSPENDED (admin action/دفعة فائتة)
  │       └──▶ REACTIVATE → ACTIVE
  │
  └──▶ CANCELLED (طلب إنهاء)
          └──▶ DELETED (بعد 90 يوم retention — بأمر صريح فقط)
```

**Provisioning Steps:**

```
1. SYSTEM → إنشاء Tenant (slug, domain, contact)
2. SYSTEM → إنشاء SubscriptionLine (line_type=base, package_version_id)
3. SYSTEM → إعادة حساب TenantEntitlement projection
4. SYSTEM → TenantModule: إنشاء records نشطة حسب الـ package
5. SYSTEM → إرسال إشعار activation
6. SYSTEM → PlatformAuditLog: CREATE_TENANT + ACTIVATE
```
